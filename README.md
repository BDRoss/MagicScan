# MagicScan

A computer-vision pipeline that identifies Magic: The Gathering cards from photos. Given a phone photograph of a card at any orientation, on a dark or light background, the pipeline returns the card name, set, and collector number by querying the Scryfall database.

## Requirements

```
pip install opencv-python numpy easyocr requests scipy
```

EasyOCR downloads approximately 100 MB of model weights on first use.

No GPU is required; the pipeline runs on CPU. Python 3.10+ recommended.

## Running

1. Place card photos (JPG or PNG) in `test_assets/`.
2. Run `python build_merged_notebook.py` to regenerate the notebook if `build_merged_notebook.py` has been edited.
3. Open `MagicScanMerged.ipynb` in Jupyter and run all cells.

Output images are written to numbered subdirectories under `output_merged/`. The final cells display a per-card diagnostic grid and a results summary table.

## Pipeline overview

1. **Card detection** — Otsu intensity threshold with luminance inversion for light backgrounds; Gaussian blur sweep for textured backgrounds; morphological close to fill interior gaps; contour filtering by area and aspect ratio.
2. **Perspective correction** — Full homographic warp (`warpPerspective`) maps the detected card quadrilateral to an axis-aligned rectangle, correcting both rotation and keystone distortion.
3. **Orientation disambiguation** — Four-rotation exhaustive search (0°, 90°, 180°, 270°) finds the content-correct orientation via Scryfall.
4. **Title detection** — EasyOCR is run on the top 20% of the corrected card; detections in the right 30% of the strip are discarded as mana cost.
5. **OCR preprocessing** — 3× bicubic upscale → non-local means denoise → unsharp mask.
6. **OCR** — EasyOCR CRNN (CNN + bidirectional LSTM + CTC decoder).
7. **Fuzzy lookup** — Scryfall `/cards/named?fuzzy=` endpoint; single-character OCR errors resolve correctly.

See `PIPELINE.md` for detailed algorithm descriptions and `COMPARISON.md` for a comparison against the original implementation.

## Known limitations

- **Severe focus or motion blur** — OCR fails when defocus blur exceeds character stroke width. Detection may succeed but the title text is unreadable. The pipeline reports the card crop and best-effort OCR text but cannot match the card.
- **Light backgrounds** — Handled by luminance inversion, but cards with large dark interior regions (level-up cards, saga frames) on light backgrounds produce more fragile contours after inversion.
- **Foil cards** — Metallic specular reflections corrupt both the threshold step and the OCR input.
- **Non-standard frames** — Showcase, borderless, and extended-art cards are generally handled. Split cards (rotated title layout) and adventure cards (two titles) are edge cases that may succeed via the four-rotation retry but are not systematically tested.
- **Non-English cards** — EasyOCR supports non-Latin scripts, but Scryfall's fuzzy endpoint is tuned for English names.
- **Parameter overfitting** — Several detection parameters (kernel sizes, blur sweep values, inversion threshold) were tuned on a small test set of 8 images. Generalisation to arbitrary photographs is not formally validated.

## Future direction

The next intended revision is a ground-up redesign targeting a genuinely general solution:
- Detection based on the card's **black border** (an invariant feature present on every card regardless of frame style or background) rather than interior brightness.
- Two-pass title extraction: first detect the title bar bounding box, then OCR on a tight crop of that box.
- Adaptive preprocessing parameters derived from image statistics rather than hardcoded values.
- Eventual deployment as an Android application written in Kotlin, calling the Python pipeline as a backend service or via a compiled model.
